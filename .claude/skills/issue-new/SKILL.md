ユーザーと自由形式の対話を通じて要望を整理し、`docs/issues/open/` に issue ファイルを作成する。

## 基本方針

- 最初に「どんなことをやりたいですか？」と聞いて、ユーザーに自由に話してもらう。
- ユーザーの言葉から必要な情報を読み取る。明らかに足りない情報があれば自然な会話の流れで聞く。
- 全項目を埋めることより、**要望の本質を正確に捉えること**を優先する。
- 十分な情報が集まったと判断したらissueドラフトを提示し、確認を求める。

## 手順

### ステップ1: 次のissue番号を決定する

最初に以下のコマンドを実行し、次の番号を決めておく（ユーザーには伝えなくてよい）。

```bash
find docs/issues -name "issue#*.md" | grep -oP '(?<=issue#)\d+' | sort -n | tail -1
```

### ステップ2: 自由対話で要望を把握する

「どんなことをやりたいですか？」と聞いてユーザーの話を聞く。

- ユーザーが話してくれた内容から、以下の情報を読み取る:
  - 何をしたいか（実施内容）
  - なぜ必要か（背景）
  - type（fix/feature/refactor/chore）
  - 完了の判断基準（acceptance_criteria）

- 明らかに不足している情報だけを、会話の流れで自然に聞く。
- 「フォームに答えてください」のような機械的な質問はしない。
- ユーザーが「まだよくわからない」と言う場合は、具体例を挙げながら一緒に考える。

### ステップ3: issueドラフトを提示して確認する

十分な情報が集まったら、issueファイルの内容全体をコードブロックで表示し、確認を求める。

「この内容でissueファイルを作成しますか？修正したい箇所があれば教えてください。」

ユーザーが修正を求めた場合は該当箇所だけ直して再提示する。
承認されたら次のステップへ進む。

以下の項目はユーザーへの確認なしに自動補完する:
- **id**: `ISSUE-{番号4桁ゼロ埋め}`
- **branch**: `{type}/issue-{番号}-{内容を英語で短く}`
- **commit_message**: `"{type}({対象機能}): {要約}"`
- **manual_checks**: 対話内容から推定（acceptance_criteriaと同じでも可）
- **paths**: 対話内容から推定（不明なら空欄）
- **notes**: 対話内容から補足事項を拾う（なければ空）

### ステップ4: issueファイルを書き出す

ファイルパス: `docs/issues/open/issue#{番号}.md`

```markdown
---
id: ISSUE-XXXX
title: {title}
type: {type}
branch: {branch}
commit_message: "{commit_message}"
paths:
  - {path}
acceptance_criteria:
  - {criteria}
manual_checks:
  - {check}
notes:
  - {note}
---

## 背景

{背景}

## 実施内容

{実施内容}

## 非対象

{非対象。なければ「特になし」}

## 使い方

### 開始

/issue-run docs/issues/open/issue#{番号}.md

### 承認後

確認OKです。
/issue-close docs/issues/open/issue#{番号}.md
```

### ステップ5: 完了報告

```
✅ docs/issues/open/issue#{番号}.md を作成しました。
実装を始めるには: /issue-run docs/issues/open/issue#{番号}.md
```

## ルール

- 実装は行わない。issueファイルの作成のみ。
- ユーザーの承認前にファイルを書き出さない。
- 対話は自然な会話スタイルで。箇条書きの質問リストを一度に投げない。
- 情報が足りなくても、推測できる場合はドラフトに反映してユーザーに確認してもらう。
- ユーザーが「もうこれでいい」「大体こんな感じ」と言ったら、足りない細部は自分で補って進める。
